What Is This Thing?

You see one website. It is actually six or seven separate programs quietly passing your request between them. Let us follow a single click.

The illusion of one app

Imagine you open the VS Platform web app, click Log in, pick your organization, and a list of employees appears. It feels like one tidy website.

It is not. Behind that smooth experience, your request hops between several small, independent programs that each do exactly one job. This module is the map of who they are.

📦
What is VS Platform, exactly?

A back-office business platform (employee and department management, the "MainTrac" product line) rebuilt as a set of small, independent programs instead of one giant one. It is built with .NET 10 and a Vue 3 browser app.

One country doctor, or a whole hospital?

The old way of building software is a monolith: one all-knowing country doctor who handles births, surgery, x-rays, and the pharmacy alone. It works, until that one doctor becomes the bottleneck for everything.

VS Platform is a microservices hospital: radiology, pharmacy, surgery, and reception each do one thing extremely well and hand the patient down the line.

🩺

Monolith — the country doctor

One program does everything. Simple to start. But to fix one thing you rebuild the whole thing, and if it falls over, everything falls over with it.

🏥

Microservices — the hospital

Many small programs, each a specialist. You can fix, update, or add staff to one department without closing the others. No single department knows everything — and that is the point.

💡
Key Insight

Giving each program one clear job — instead of one program doing all jobs — is one of the most important ideas in software. Engineers call it separation of concerns.

⚠️
The hidden cost

Splitting into many programs means the network between them is now part of your app — and a network can be slow, or fail. Most of this course is about taming that wiring.

This is real code, not magic

One nice trick keeps all this manageable: the same built browser app can be pointed at the test, staging, or live environment without rebuilding it. It reads its settings when it starts up. Here is the actual code that does it.

CODE 
function readConfig(): AppConfig {
  const fromWindow = typeof window !== 'undefined' ? window.__APP_CONFIG__ : undefined;
  if (!fromWindow) return FALLBACK;

  return Object.freeze({
    apiBaseUrl: fromWindow.apiBaseUrl || FALLBACK.apiBaseUrl,
    services: Object.freeze({ ...fromWindow.services }),
    featureFlags: Object.freeze({ ...fromWindow.featureFlags }),
    env: fromWindow.env || FALLBACK.env,
    auth: Object.freeze({ /* ... */ }),
  });
}
export const runtimeConfig: AppConfig = readConfig();

Define a little routine that figures out the app's settings.

Look at the web page for a settings object the server dropped in (window.__APP_CONFIG__) — if we are even running in a browser.

If nothing was provided, fall back to sensible built-in defaults.

Otherwise, hand back a locked-down copy of those settings.

Where to send data requests (the address of the Business API).

The addresses of the other services.

Which optional features are switched on.

Which environment this is — test, staging, or live.

The login settings (details hidden here for brevity).

End of the settings.

Run that routine once at startup and remember the answer.

The key word is Object.freeze: once the settings are read, they are locked so nothing can quietly change them later. That single habit is why one built file can safely run in three different environments.

03

The Login Relay

You logged in once. So why does your browser swap passes three or four times before the employee list even loads? Because a token is a wristband — and every door wants its own.

One login, many doors

Picture a music festival. You buy one ticket. At the gate, you trade it for a paper wristband. From then on, nobody asks for your ticket again — the wristband is your proof.

VS Platform works the same way. You log in once. After that, the platform hands your browser little digital wristbands called tokens. Each request to the system shows a token instead of your password.

🎟️
The core metaphor

Ticket = your login. Wristband = a token. The pass office that upgrades your wristband for a specific stage = the STS. Hold onto these three — the whole module rides on them.

What is actually printed on a wristband?

A token is usually a JWT — a small bundle of facts the system can read and trust because it is digitally signed. Here is what a VS Platform wristband carries:

sub Who you are — your identity, like the name on a festival ticket.
aud The audience — the single door this wristband is allowed through. A pass stamped for the gateway is useless at the Business API.
exp When it expires — wristbands are short-lived on purpose, often just a few minutes.
org_id Which organization you are acting as right now — the festival you bought into (VS Platform serves many companies at once).

The key idea: a token is not a master key. It is a wristband stamped for one door, for a short time. That single design choice drives everything that follows.

The three security characters

Three services run the whole pass operation. Meet the cast — each has one job, and keeping the jobs separate is what makes the system safe.

🎫
Keycloak — the box office

Keycloak is where you actually log in. It checks your password once and issues your original ticket. This is the only place authentication (proving identity) happens.

🏢
STS — the pass office

The STS is the pass office. It takes your wristband and stamps a new one aimed at exactly one door. It is the only service that holds the secret keys for these trades.

📋
Authz — the rulebook

The Authz service answers "is this person allowed?" — the org context and permissions. We meet it fully in Module 5; here it is just the source of your org_id.

⚠️
The naive move AI loves to suggest

Ask an AI to "call the API from the frontend" and it will often paste a secret key straight into the browser. The browser is fully visible to anyone — that is like leaving the master wristband stamp on the festival gate. Notice that in VS Platform the browser holds zero secrets; only the STS does. Steer your AI toward this pattern.

Reading the real code

Now the payoff — the exact code that runs this relay. First, the browser side: before calling anything, it makes sure it is holding a gateway-stamped wristband.

CODE · auth.store.ts 
async function ensureGatewayEntryToken(): Promise<void> {
  if (!oidcUser.value?.access_token) return;
  if (hasGatewayAudience(oidcUser.value.access_token)) return;

  const subjectToken = oidcUser.value.access_token;
  persistKeycloakSubjectToken(subjectToken);

  const gatewayEntry = await exchangeForGatewayEntry(subjectToken);
  await applyAccessToken(gatewayEntry.access_token, gatewayEntry.expires_in);
}
PLAIN ENGLISH

A reusable routine: "make sure we are holding a gateway wristband."

If we are not even logged in yet, there is nothing to do — stop here.

If our current wristband is already stamped for the gateway, we are done. No need to trade.

Otherwise: keep a copy of our original login ticket from Keycloak...

...and stash it safely so the relay can use it.

Walk over to the pass office and trade the login ticket for a gateway wristband.

Start using that new gateway wristband, and remember when it expires.

Now the gateway side. For every single request, the gateway upgrades the caller's pass into one aimed at exactly the destination door.

var orgId = httpContext.User.FindFirst("org_id")?.Value;
var downstreamScopes = ResolveDownstreamExchangeScopes(httpContext);
var exchanged = await tokenBroker.GetTokenAsync(
    new TokenRequest
    {
        Audience = route.TargetAudience,
        Scopes = downstreamScopes,
        OrganizationId = orgId
    },
    httpContext.RequestAborted);

if (exchanged.Error is { } exchangeError)
{
    await VsAuthProblems.WriteProblemAsync(
        httpContext,
        exchangeError.ToStatusCode(),
        "Token exchange failed",
        exchangeError.Message,
        exchangeError.Reason);
    return;
}

httpContext.Items[ExchangedTokenItemKey] = exchanged.Token.Value;
await next();
PLAIN ENGLISH

Grab the token broker — the helper that talks to the pass office.

Read which organization the user is acting as, from their wristband.

Figure out which permissions (scopes) this downstream door needs.

Ask the pass office for a fresh token...

...described by a request that says:

stamp it for exactly this destination door (Audience),

grant only these permissions,

and tie it to this organization.

(...and bail out cleanly if the user cancels mid-flight.)

 

If the trade failed for any reason...

...send the caller a clear, structured error...

...with the right status code (like 401 or 403)...

...a short title, the detail, and the reason it failed...

...then stop — no silent break, no half-finished request.

 

Success: tuck the new pass aside to attach when forwarding...

...and continue handing the request down the line.

💡
Why exchange instead of one all-powerful token?

A pass scoped to one door cannot open the others. If it leaks, the blast radius is exactly one service — not your whole account. Engineers call this design least privilege, and the relay enforces it automatically on every request.

04

The Gateway — One Front Door

How a single public address quietly delivers your requests to five different services behind the scenes.

One address, many rooms

Your browser only ever talks to one address. Yet behind that address sit five separate services. So how does one door lead to many rooms?

Meet the gateway — built on a piece of Microsoft technology called YARP, a reverse proxy.

📮
The metaphor: a postal sorting facility

Every package arrives at one loading dock — the mailroom. A clerk reads the label, re-labels it for the right department, stamps it "came through the official mailroom," and sends it on. The departments never deal with the public directly. Only the mailroom does.

Two ways to wire up many services

🚫

Browser talks to everyone

The browser would need five addresses, five logins, and five CORS permission grants. A security and configuration nightmare.

✅

Browser talks to one door

The browser knows a single address. The gateway handles the routing, the security checks, and the relaying internally — where no public browser rules apply.

Anatomy of a route

How does the clerk know where a package goes? It follows a route — a recipe that matches part of the incoming address and names a destination.

In the code, one route is a small record. Here is what it declares:

CODE 
public sealed record GatewayRoute(
    string PathPrefix,
    string DownstreamPrefix,
    string TargetAudience,
    string TargetBaseUrl,
    bool AnonymousForwardingAllowed = false,
    bool PreservePath = false)
{
    public IEnumerable<RouteConfig> ToYarpRoutes(int order, bool rateLimitEnabled)
    {
        var routeId = ToStableId(TargetAudience);
        var pathPrefix = NormalizePathPrefix(PathPrefix);
        yield return ToYarpRoute(routeId + "-exact", pathPrefix, order * 2, rateLimitEnabled);
        yield return ToYarpRoute(routeId + "-catch-all", pathPrefix + "/{**catch-all}", order * 2 + 1, rateLimitEnabled);
    }
}
PLAIN ENGLISH

One route is a recipe with a few ingredients.

The prefix to match, like /api.

The prefix to add on the way out (more on that next screen).

The audience — who the access pass should be made out to.

The actual internal address to forward to.

Whether to allow requests with no login through.

Whether to leave the path untouched instead of rewriting it.

From this one recipe it generates two real rules...

...one for the exact path (e.g. /api on its own)...

...and one catch-all for everything underneath it (e.g. /api/employees).

PathPrefix The label the clerk reads on the way in — "anything starting with /api"
TargetAudience Who the access pass gets stamped for — so only the right department accepts it
TargetBaseUrl The internal address of the department that actually handles the package

404 vs 403 — a debugging superpower

404 means "no route" — a routing problem. 403 means "no permission" — a security problem. Knowing which one you got tells you instantly whether to look at the gateway's routes or at the caller's access pass.

Self-updating routes — the clever trick

Here is the part that feels like magic. When a new service is deployed, the gateway can learn its route while still running — no restart needed.

It watches a ConfigMap in Kubernetes. A new service announces its route there; the gateway notices and hot-reloads it instantly.

05

Who Can Do What

Authorization & multi-tenancy — the keycard that decides which doors open, and how one building keeps each company's offices private.

The front door is not the whole building

Two people log in successfully. One sees the "Delete employee" button; the other doesn't. Same building, different keycards. So who decides — and how does the platform keep Company A's data invisible to Company B?

Picture a large office building. Getting past the lobby is one thing. What floors and rooms your keycard opens is a completely different thing.

🚪
Authentication — the lobby

Proving who you are. Covered in Module 3: you trade your login for a wristband-style pass. This only gets you through the front door.

🪪
Authorization — the keycard

Deciding what you may actually do. Same building, but your keycard opens the rooms for your job and no others. This module is all about the keycard.

💡
Two different questions

Authentication asks "who are you?" Authorization asks "is this allowed?" They sound similar but they are separate steps — confusing the two is one of the most common ways security goes wrong.

The keys on the keycard

A keycard is just a list of permissions — named rights, each one a door it can open. VS Platform names them in plain, readable strings so both humans and code can reason about them.

employees.read View employee records — like peeking into the HR file room without touching anything.
employees.write Create, edit, or delete employees — the keycard that actually changes the files.
departments.read View the org chart and department list — look, but don't rearrange.
departments.write Add, rename, or remove departments — restructure the floor plan.
gateway.access The coarse "may enter the building at all" pass. Module 4's front-gate check — necessary, but nowhere near sufficient.
authz.orgs.read Read which organizations and memberships exist — a management key for the building directory itself.
🔑
Coarse vs fine

Module 4's gateway only checked gateway.access — "is this person allowed in the building?" This module is the fine-grained decision: which specific room, in which specific company's office.

The keycard check, in real code

Here is an endpoint that lets you read your organizations. Notice it has two locks: a fast gate before the code runs, and an authoritative check inside, pinned to one specific company.

public async Task<IActionResult> GetMine(string orgId, CancellationToken cancellationToken)
{
    var subject = User.FindFirstValue("sub") ?? callerAccessor.Current.Subject;
    if (string.IsNullOrWhiteSpace(subject))
    {
        return Unauthorized();
    }

    var decision = await authorization.AuthorizeAsync(
        AuthzPermissions.OrgsRead,
        ResourceContext.For("organization", orgId).InOrganization(orgId),
        cancellationToken);
PLAIN ENGLISH

This handles a request to "read my organizations."

Lock #1: Before any code runs, the [RequireVsPermission] tag turns anyone away who lacks the OrgsRead key. No key, no entry — the room never even opens.

The handler runs, asking for one input: which org you're after.

Figure out who's asking by reading the token claim called sub — the unique ID stamped on the pass.

If we can't tell who's asking, stop right here.

Lock #2: Now the authoritative check. AuthorizeAsync doesn't just ask "do they have the key?" —

— it asks for the OrgsRead permission specifically InOrganization(orgId). So even a valid user can't peek into a company they don't belong to.

It hands back a decision — a verdict, not a crash. (We open that up on the next screen.)

That second lock — org-scoping — is the quiet hero. The key alone isn't enough; the key has to be valid for that company's floor.

Claimed vs Authorized — the lesson that matters most

VS Platform deliberately uses two tiers of checking. They look similar, but mixing them up is how data leaks. Read this contrast slowly.

👀

Claimed — a fast peek

Reads the token claim already in your pass. Quick, no server round-trip. Good for hiding a button the user can't use. It's a hint, not a guarantee — the pass was stamped earlier and only the browser is trusting it.

🛡️

Authorized — the real verdict

Asks the authoritative authorization check, pinned to the org. Required before any change: delete, edit, create. This is the door that physically refuses to open. Slower, but it's the truth.

⚠️
NEVER gate a change on a claim alone

Hiding the "Delete" button is a courtesy, not a lock. Anyone can call the API directly and skip your hidden button entirely. The server must always run the authoritative, org-scoped check before it actually changes anything. A peek is for the eyes; a verdict is for the lock.

A verdict, not a crash

When the platform decides, it returns a small record describing the outcome, instead of throwing an error from deep inside. That makes the answer easy to log, explain, and handle the same way everywhere.

CODE 
public sealed record AccessDecision
{
    public required bool IsAllowed { get; init; }
    public required string Permission { get; init; }
    public AccessDenyReason DenyReason { get; init; }
    public string? GrantSource { get; init; }
    public string Detail { get; init; }

    public static AccessDecision Allow(string permission, string grantSource, ...) { ... }
    public static AccessDecision Deny(string permission, AccessDenyReason reason, ...) { ... }
    public void ThrowIfDenied() { ... }
}
PLAIN ENGLISH

A little "report card" describing one access decision.

IsAllowed: the bottom-line yes or no.

Permission: which key was being checked (e.g. employees.write).

DenyReason: why it was refused — wrong floor? missing key? — instead of a silent "no."

GrantSource: if allowed, where the permission came from (which role gave you the key).

Detail: a human-readable note for logs and debugging.

Two tidy factory methods: build an "Allow" verdict or a "Deny" verdict with its reason.

And if a caller does want a hard stop, it can opt in with ThrowIfDenied() — its choice, not the default.

Returning a verdict means every service turns the same answer into the same clean error, with the same auditable reason. The check doesn't surprise you by crashing somewhere unexpected.

One building, many companies

Several companies share the same building. Multi-tenancy is exactly that: one running system serving many customers (called tenants), each kept walled off from the rest.

VS Platform does this with schema-per-organization: inside a single Postgres database, each org gets its own schema — a labeled compartment with its own tables.

🗄️ ONE Postgres database
🔒 org_alpha

Company Alpha's employees, departments, and records. Sealed in its own compartment.

🔒 org_beta

Company Beta's employees, departments, and records. Sealed in its own compartment.

🪪 A keycard stamped for Beta opens org_beta only.

It will not open org_alpha — the org-scoped check refuses at the door.

🚨
This is the scariest class of bug

If a request from Alpha ever returns Beta's data, that's a cross-customer data leak — the failure no business can afford. Org-scoping (InOrganization) is the wall. Never weaken it for convenience.

Keeping the directory and the lobby in sync

The building directory holds the real, authoritative answer for who's allowed where. The lobby system just needs a faithful copy so it can stamp passes quickly. Something has to keep the copy honest.

1
The authz Postgres database is the single source of truth

The authz service owns the rules — orgs, who belongs to them, and what they may do. If anything disagrees with this, this wins.

2
Keycloak is a fast copy, not the authority

So that login passes carry the right org context, the rules are mirrored into Keycloak — the lobby. But Keycloak is a projection of the truth, never the truth itself.

3
A reconciler projects one into the other

A background job continuously copies the authz rules into Keycloak. It runs single-flight — never two at once — so the copy converges on the truth without trampling itself.

📌
Single source of truth

Postgres (owned by authz) is canonical; Keycloak is a synced copy kept fresh by the reconciler. When you wonder "what's really allowed?", the answer lives in the authz database — not the login pass. The pass is just yesterday's photocopy of today's directory.

Scenario 2

Shared Foundations

Toolkits everyone reuses, and the flight simulator that boots the whole platform on one laptop.

Don't repeat yourself, six times

Six services, three databases, a login server. How does one developer run all of that without losing their mind — and why don't six teams each write their own login code? Two answers: shared toolkits and a flight simulator.

🔌
Metaphor: standardized plugs and sockets

Instead of every service inventing its own way to plug into login, databases, and monitoring, they all use the same shared "outlets." Change the outlet once, and every appliance benefits.

These outlets are NuGet packages — six of them, living in a repo called vs-plat-packages. Each one handles a cross-cutting concern: a job every service needs but none should reinvent.

🔐

Auth

The main login + permission toolkit. Wires a service up to check "who are you, and are you allowed?" the same way everywhere.

🧩

Auth.Abstractions

The shared "shape of the socket." Defines the rules every login backend must follow, without saying which one is plugged in.

🗝️

Auth.Keycloak

One plug-in login backend: get passes from Keycloak, the real identity server.

🎟️

Auth.Sts

A different plug-in login backend: get passes from the STS (Security Token Service) instead.

🗄️

Data

The shared database toolkit — so every service talks to Postgres with the same rules, connections, and conventions.

⚙️

ServiceDefaults

The starter pack: monitoring, health checks, retries — turned on for every service with a single line of code.

The plug/socket idea pays off most for login. Swapping the entire login backend is one line: UseKeycloakTokenBroker() to use Keycloak, or UseStsTokenBroker() to use the STS. The service code never changes — only which plug is in the socket.

One call, all the plumbing

Every service starts its life with the same single line: AddServiceDefaults(). That one call flips on the whole "starter pack" — monitoring, health checks, and resilient network calls — so no team has to wire it by hand.

CODE 
public static TBuilder AddServiceDefaults<TBuilder>(this TBuilder builder) where TBuilder : IHostApplicationBuilder
{
    builder.ConfigureOpenTelemetry();
    builder.AddDefaultHealthChecks();
    builder.Services.AddServiceDiscovery();

    builder.Services.ConfigureHttpClientDefaults(http =>
    {
        if (builder.Configuration.GetValue("HttpClient:StandardResilience:Enabled", true))
        {
            http.AddStandardResilienceHandler();
        }
        http.AddServiceDiscovery();
    });

    return builder;
}
PLAIN ENGLISH

This is the one setup recipe every service runs at startup.

Turn on OpenTelemetry so we can see what the service is doing.

Add health checks so the system knows the service is alive.

Turn on service discovery so services find each other by name.

Now set sensible defaults for every outgoing network call this service makes.

Unless someone explicitly turned it off in configuration...

...add automatic retries, so a flaky network blip doesn't crash a request.

Also let those calls use service discovery to reach their target.

Hand the fully-equipped service back, ready to run. Written once, used everywhere.

What's telemetry buying us? When something is slow, you can see where. When a service dies, the system knows. When the network hiccups, the request quietly retries instead of failing. All from one line — written once in the shared package, reused by everyone.

The pluggable token broker

Back in the login flow, a service constantly needs a "pass" (a token) to talk to other services. But here's the trick: the service shouldn't care how the pass is obtained.

That's the job of the interface called ITokenBroker. It defines the socket; Keycloak and STS are two different plugs that both fit it.

📜
ITokenBroker — the contract

"Whoever you are, you must be able to hand me a valid pass when I ask." That's all the service knows about.

🗝️
Keycloak plug — UseKeycloakTokenBroker()

Gets passes straight from Keycloak, the real identity server. Used when running against the full login stack.

🎟️
STS plug — UseStsTokenBroker()

Gets passes from the Security Token Service instead. Same socket, different supplier.

💡
Key Insight: fix it once

Separating shared behavior into shared libraries gives you consistency and "fix-once." A bug in how tokens are cached lives in one package — fix it there, and all six services get the fix. No hunting through six codebases.

The flight simulator

✈️
Metaphor: a flight simulator

The orchestrator is a faithful replica of the whole platform that runs on your laptop — same cockpit, same instruments, no real plane needed. The local simulator mirrors production wiring, so "works locally" means a lot more here than usual.

That orchestrator is .NET Aspire, living in vs-plat-orchestrator. Its centerpiece is the AppHost: one program that lists every piece and how it connects. Type one command — aspire run — and the whole stack boots, in the right order.

The order matters: a service can't start before the thing it depends on is ready. The simulator enforces this with WaitFor.

1
Postgres (the database) starts

The warehouse where everything is stored comes online first — almost everyone depends on it.

2
Redis (the fast cache) starts

Redis comes up next so services have somewhere quick to stash short-lived data.

3
Keycloak (the login server) starts

The identity server boots and gets ready to issue passes.

4
STS (the token service) starts

The token booth comes online, ready to mint passes for services that ask.

5
The reconciler job runs

A one-time setup job syncs permissions into place. The API waits for this to finish before starting.

6
Authz (the permission service) starts

The service that answers "is this person allowed?" comes online.

7
The Business API starts

The actual app logic boots — only after its database, cache, and login pieces are all ready.

8
Gateway + web client start

The traffic router and the Vue browser app come up last — now there's a working platform behind them to serve.

Handing out addresses, automatically

Here's how the AppHost describes one piece — the Business API — as a recipe. Notice it never hardcodes an address; it asks for each dependency's address at runtime.

CODE 
api = builder.AddContainer("vsplatform-api", apiOptions.Image, apiOptions.Tag)
    .WithImagePullPolicy(ImagePullPolicy.Missing)
    .WithHttpEndpoint(targetPort: 8080, name: "http")
    .WithEnvironment("ASPNETCORE_ENVIRONMENT", "Development")
    .WithHttpHealthCheck("/health", endpointName: "http")
    .WithExternalHttpEndpoints()
    .WithReference(database).WaitFor(database)
    .WithReference(redis).WaitFor(redis)
    .ApplySharedAuthEnvironment(realm)
    .WithEnvironment("VSPlatformAuth__Audience", "vs-platform-api")
    .WithEnvironment("VSPlatformAuth__Broker", "Sts")
    .WithEnvironment("VSPlatformAuth__Sts__BaseUrl", sts.GetEndpoint("http"))
    .WaitForCompletion(authReconciler);
PLAIN ENGLISH

Define the Business API by running its container image (a sealed, ready-to-run package).

Only download the image if we don't already have it locally.

It listens on port 8080 — name this doorway "http".

Run it in "Development" mode (the simulator setting).

Check it stays healthy by polling its /health endpoint.

Allow it to be reached from outside the simulator too.

It needs the database — and must wait until the database is ready.

It needs Redis — and must wait for Redis too.

Apply the shared login settings (the standardized plug from earlier).

Tell it which audience its passes are for.

Tell it to use the STS plug for getting passes.

Hand it the STS's address — fetched live, never hardcoded.

And don't start until the permission-sync job has fully finished.

The magic line is sts.GetEndpoint("http"): the API is handed the STS's real address at runtime. On your laptop that's one address; in the cloud it's another — same code, no edits.

Source mode vs container mode

The simulator has a clever switch for each service: run it from your local source code, or run the pre-built image the team already shipped. You choose per service.

🛠️

Source mode

Runs the service straight from code on your machine. Edit, save, see your change. Perfect for the one service you're actively working on.

📦

Container mode

Runs the stable, shipped image (the code you saw above). Perfect for the five services you just need running in the background.

So you can edit one service from source while pulling rock-solid published versions of the rest — fast, focused, and you never break the others.

☁️
This same model ships to production

The "describe each piece + hand it its dependencies" idea maps straight to AKS (Azure Kubernetes) via Helm. Same wiring, real cluster. That's Module 7.

Shipping It Safely

From your laptop to live users — the assembly line that inspects every change before it reaches anyone.

The belt between "push" and "live"

You changed one line and pushed it. Minutes later it's running for real users — or it's blocked with a red X. So what happens on the belt in between?

Picture a factory assembly line. Your code change is a part placed on a moving belt. It rolls past a series of inspection stations, and only a part that passes every station gets boxed up and shipped to the store shelf. This whole automated belt has a name: CI/CD.

🏭
The whole metaphor in one sentence

A code change is a part on a belt. Inspection stations check it. If any station fails, the belt stops and the part never reaches customers. That's the entire idea of CI/CD.

The assembly line, station by station

Here's the actual belt VS Platform runs every time you push. Follow a single change from left to right.

1

Push

→

3

→
4

→

A quick glossary for three of those stations, since they'll come up again:

🔨
Build

Turning human-written source code into a program a computer can actually run. Like assembling raw parts into a finished gadget.

🧹
Lint

An automatic quality inspector that flags messy or suspicious code before it goes further — a spell-checker for code.

✅
Test

Pre-written checks that run the code with known inputs and confirm it produces the expected results. The robots try it before any human does.

Write the recipe once, run it everywhere

A factory doesn't reinvent its inspection procedure on every belt. It writes one master procedure and posts it at every station. VS Platform does the same: the steps for the belt live in one place, a repository called vs-plat-shared-pipelines.

Every other repo just imports that shared recipe instead of writing its own. The recipe — the ordered list of belt stations — is called a pipeline. And crucially, the import is frozen to a specific version tag.

CODE 
definitions:
  imports:
    dotnet: vs-plat-shared-pipelines:v1.3.0:.bitbucket/dotnet-pipelines.yml

pipelines:
  default:
    import: dotnet-verify@dotnet
  branches:
    master:
      import: dotnet-verify-publish-deploy@dotnet
  pull-requests:
    '**':
      import: dotnet-verify@dotnet
PLAIN ENGLISH

This repo doesn't write its own belt — it pulls the steps in from the shared-pipelines repo...

...and freezes them to version v1.3.0. The recipe can't change underneath you by surprise.

For any everyday push, run "verify" — that's build plus test.

But on the master branch (the official live version), run the full "verify-publish-deploy" — build, test, and package and ship.

On a pull request (a proposed change up for review), run "verify" too — prove it's sound before anyone merges it.

Two new terms in that translation worth pinning down:

master The branch that represents the official, live version of the code. Pushing here means "this is real now."
pull-request A proposed change put up for review and automated checks before it's allowed to join master.
💡
Why one shared recipe matters

Fix the build process once and every repo gets the fix. No repo drifts off with its own slightly-different belt. Engineers call this "single source of truth" — and it's why your AI assistant can reason about any repo the same way.

Packaging: the shipping container & its labels

Once a change passes inspection, it gets boxed up for shipping. The box is a container image: a sealed box holding the program and everything it needs to run, so it behaves identically on any machine.

The factory builds the box once, then slaps multiple labels on the same box so different people can find it different ways. Some labels stay glued to one exact box forever — they're immutable. One label gets peeled off and re-stuck on whatever the newest box is.

🏷️

master — moving

Always points at the newest official box. Peeled off and re-stuck each release. Means "give me the latest." Never trust it to be the same box twice.

🔒

<short-sha> — immutable

A label made from the change's unique fingerprint. Glued to one exact box forever. Perfect for pinning to — or rolling back to — a precise build.

🔢

<branch>-<build#> — immutable

Names which branch it came from and which build number. Also frozen to one box — handy for tracing exactly which run produced it.

🔁
Moving vs. immutable, in one breath

A moving tag (master) answers "what's newest?" An immutable tag (the SHA) answers "give me that exact build, no matter how much time has passed." You want both — speed and the ability to rewind.

The end-to-end gate: a robot opens your app

The most important inspection station is the end-to-end (E2E) gate. Unit tests check each part in isolation; this gate checks the assembled machine. It's a merge gate: if it's red, the change cannot merge. Period.

It works by running a smoke test using Playwright, which steers a real but invisible headless browser. Here's the heart of the test.

CODE 
[Fact]
public Task AspireStack_ReturnsOk() => RunTestAsync(async () =>
{
    var page = await _context.NewPageAsync();

    var response = await page.GotoAsync(
        aspire.ClientBaseUrl, new() { WaitUntil = WaitUntilState.Commit });

    Assert.NotNull(response);
    Assert.True(response.Ok, $"Expected HTTP 200, got {response.Status}");
});
PLAIN ENGLISH

[Fact] marks this as one automated test the robot must run.

Open a fresh, invisible browser tab.

Navigate that tab to the running web app's address...

...and wait until the page has actually started loading.

Insist the app answered at all (not silence).

And insist the answer was "OK" (HTTP 200). If not, fail loudly — and the merge is blocked. A robot clicks your app before any human does.

What the gate actually does, step by step

1
Clone the orchestrator

Grab the project that knows how to boot every service together — the "conductor" of the whole platform.

2
Boot the full stack in container mode

Start every service from the freshly-shipped container boxes — the real thing, assembled, not mocked.

3
Open the app in Playwright

Launch the invisible headless browser and point it at the running web app, just like a real visitor.

4
Run the smoke tests

Confirm the page comes back OK. Does the whole machine even switch on without smoke?

5
Pass or fail the merge

Green light → the change may merge. Red light → the belt stops and the change is blocked.

🛩️
Remember the flight simulator?

This gate reuses the exact same Aspire stack you ran on your laptop in Module 6 — but in container mode, in the cloud. The thing you used to develop is the thing that now guards the door.

Delivery: onto the live shelves

A box that clears every station gets delivered to the store shelf — the live cloud cluster. VS Platform runs on Kubernetes, specifically the managed flavor called AKS.

Rather than placing each box by hand, the line uses Helm to install and upgrade everything in one move. And the last station is automatic: every push to master triggers a Helm upgrade onto the dev cluster — no human needed.

🚚
Helm upgrade

One command swaps the running app for the new box, with all its settings — like a forklift loading a fully-labeled pallet onto the shelf.

⚙️
AKS cluster

The cloud warehouse that keeps the boxes running, restarts them if they crash, and serves real users.

🤖
Auto-deploy on master

Push to master → the belt rebuilds, re-gates, repackages, and ships to the dev cluster on its own.
